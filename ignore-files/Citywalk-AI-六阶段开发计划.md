# Citywalk AI 开发计划

> 计划版本：v1.0  
> 推荐阶段数：6  
> 目标：以“完善模式 Complete”为第一条主线，先完成可真实使用的路线闭环，再逐步增加 AI、主题和亲友功能。

---

## 0. 开发原则

每个阶段必须满足：

1. 有可运行、可演示的成果；
2. 有明确验收标准；
3. 不依赖下一阶段才能使用；
4. 外部 API 有 mock 或缓存，避免开发测试持续消耗额度；
5. 数据库变更使用 migration；
6. 重要纯函数有测试；
7. 每阶段结束都用真实城市路线走查。

建议先用 3～5 个城市区域作为固定测试样本：

- 上海武康路 / 思南路；
- 南京总统府 / 先锋书店周边；
- 北京鼓楼；
- 广州沙面；
- 杭州湖滨。

# 阶段 1：项目基础与视觉骨架

## 目标

搭好可持续开发的工程、部署、设计系统和静态页面骨架。

## 开发内容

### 工程

- 创建 Next.js App Router + TypeScript 项目；
- 配置 `output: "export"`；
- 配置 GitHub Actions 和 GitHub Pages；
- 配置自定义域名适配（可选）；
- 建立环境变量模板；
- 配置 ESLint、Prettier、TypeScript strict；
- 配置 Vitest；
- 配置 Playwright 基础环境。

### UI

- 建立颜色、字体、间距和组件 Token；
- 实现顶部导航；
- 实现桌面 / 手机响应式布局；
- 实现首页；
- 实现三种模式入口；
- 实现 AI 规划页静态骨架；
- 实现 City Reader 静态骨架；
- 实现阅读 / 导航模式切换的 UI 状态。

### 数据

- 定义核心 TypeScript 类型；
- 建立 URL 适配层；
- 建立 Repository 接口；
- 实现 localStorage 草稿。

## 阶段成果

- 网站可通过 GitHub Pages 打开；
- 首页、规划页、路线页三张核心页面可浏览；
- 手机与桌面响应式有效；
- 可以创建和恢复一份假数据路线草稿。

## 验收标准

- `npm run build` 可稳定生成静态文件；
- 直接刷新 `/route/?id=demo` 不报错；
- 390px 和 1440px 页面无明显布局问题；
- Lighthouse 基础可访问性无严重错误；
- 路由 URL 不由组件自行拼接。

# 阶段 2：Supabase 数据层与用户保存

## 目标

建立真实数据模型、登录、RLS、保存和分享基础。

## 开发内容

### Supabase

- 创建 Supabase 项目；
- 配置本地 CLI；
- 建立 migrations；
- 创建：
  - profiles
  - places
  - routes
  - route_stops
  - route_constraints
  - route_snapshots
  - route_shares
- 启用 RLS；
- 编写 owner/viewer 基础策略；
- 创建 Storage bucket（先预留，不必立即上传图片）。

### 前端

- 接入 Supabase Auth；
- 支持邮箱魔法链接或 OTP；
- 未登录使用本地草稿；
- 登录后将本地草稿迁移到账号；
- 实现“我的路线”；
- 实现保存、复制、删除；
- 实现只读分享页；
- 实现分享撤销和过期。

### 安全

- 分享读取通过 Edge Function；
- 匿名用户不直接读取 routes；
- 服务端密钥只存在 Edge Function；
- 设置 CORS 白名单。

## 阶段成果

- 用户可以未登录创建草稿；
- 登录后保存到云端；
- 可以生成分享链接；
- 手机端能打开分享路线。

## 验收标准

- A 用户不能读取 B 用户私有路线；
- 撤销后分享链接立即失效；
- 登录迁移不会重复创建路线；
- 分享接口不返回 owner 的邮箱等信息；
- 所有 public 表启用 RLS。

# 阶段 3：高德地点与路线内核

## 目标

不依赖 AI，先完成真实可走的路线。

## 开发内容

### 高德

- 配置高德 JS API；
- 配置高德 Web 服务代理；
- 实现地点输入提示；
- 保存高德 POI ID、地址、行政区和 GCJ-02 坐标；
- 实现手工地点；
- 实现步行路线规划；
- 支持公交 / 骑行作为后续开关，第一版可只开放步行；
- 地图绘制站点和路线；
- 实现“打开高德导航”。

### 路线内核

- 起点；
- 终点；
- 必去点；
- 固定预约时间；
- 停留时间；
- 简单排序；
- 时间轴计算；
- 总距离和总时长；
- 约束冲突提示；
- 拖拽排序后重算；
- 添加 / 删除站点后重算。

### 测试

- 时间轴纯函数测试；
- 固定预约冲突测试；
- 重复 POI 测试；
- 起终点测试；
- 高德失败降级测试。

## 阶段成果

用户可以输入几个真实地点，得到一条地图路线和时间轴，并手动编辑。

## 验收标准

- 所有地图坐标明确标记 GCJ-02；
- AI 未接入时产品仍可完成基础路线；
- 不把估算时间伪装成高德真实时间；
- 拖拽站点后能重新计算；
- 冲突时给出明确原因和建议。

# 阶段 4：沿途智能补点与 Complete 模式

## 目标

完成产品的第一项差异化核心能力。

## 开发内容

### 候选点搜索

- 对路线折线采样；
- 对采样点做周边 POI 搜索；
- 按 POI ID 去重；
- 支持地点类型筛选：
  - 景点
  - 博物馆
  - 历史建筑
  - 书店
  - 咖啡馆
  - 餐厅
  - 公园
- 对 Top N 候选计算真实绕行时间；
- 保存 route_candidates；
- 候选点缓存。

### 候选点评分

- 绕行成本；
- 主题匹配；
- 兴趣匹配；
- 类型多样性；
- 时间窗口；
- 重复惩罚；
- 预约风险。

第一版采用明确的规则权重。

### UI

- 沿途可选点面板；
- “非常顺路 / 推荐 / 可考虑”分类；
- 展示新增交通时间和停留时间；
- 加入路线；
- 设为备用；
- 忽略；
- 加入后重新计算；
- 显示对结束时间和预约的影响。

## 阶段成果

用户提供 2～4 个必去点后，系统可以提供真实沿途候选点。

## 验收标准

- 候选点均来自真实 POI；
- 不自动静默加入路线；
- 加入候选点后时间轴准确变化；
- 路线段绕行计算有测试；
- 同一 POI 不重复出现；
- API 调用数量有上限和缓存。

# 阶段 5：AI 路线协作与轻主题

## 目标

让 AI 提升规划和阅读体验，但不成为事实来源。

## 开发内容

### AI 服务

- 接入 DeepSeek；
- 实现 `parse-intent`；
- 实现候选点排序与推荐理由；
- 实现路线标题和摘要；
- 实现站点短介绍；
- 使用 JSON 模式；
- 使用 Zod 校验；
- 实现一次修复重试；
- 记录 prompt 版本、token、耗时和费用；
- 增加用户每日额度和项目费用上限。

### Complete 模式

- 自然语言输入转结构化需求；
- AI 对真实候选点排序；
- AI 解释取舍；
- 生成路线变更摘要；
- 不允许 AI 创造新 POI。

### 轻主题

第一版：

- 历史；
- 文学。

支持：

- 仅改变讲解；
- 为候选点增加主题匹配分；
- 站点短介绍；
- 来源待确认标识；
- 单站“展开深读”按需生成。

### 失败降级

- AI 失败时保留地图路线；
- 使用规则排序；
- 使用模板推荐理由；
- 主题内容可稍后生成。

## 阶段成果

用户能用自然语言完善路线，并阅读有历史或文学气质的站点内容。

## 验收标准

- AI 输出不能包含数据库中不存在的 POI ID；
- AI 失败不阻塞保存和分享；
- 未核验历史内容有明确标识；
- 单条路线 AI 成本可统计；
- 相同请求有幂等保护；
- 提示词或 Schema 变化有版本记录。

# 阶段 6：打磨、真实使用与下一阶段决策

## 目标

把产品交给本人和亲友真实使用，以数据决定是否继续扩展。

## 开发内容

### 产品打磨

- 手机端导航模式；
- 下一站卡片；
- 地图全屏；
- 临时跳过；
- 临时替换；
- 路线快照与恢复；
- 复制朋友路线；
- 错误状态和空状态；
- 性能优化；
- 图片懒加载；
- 地图懒加载；
- PWA 基础能力（可选）。

### 可观测性

- 结构化日志；
- 高德调用量；
- AI token 和费用；
- 生成成功率；
- 候选点加入率；
- 分享率；
- 第二次打开率；
- 错误报警。

### 真实测试

- 至少完成 10 条路线；
- 至少实际走完 3～5 条；
- 收集亲友反馈；
- 记录路线不合理原因；
- 记录哪些内容真正被阅读；
- 记录手机端途中操作痛点。

### 决策评审

根据数据决定下一步优先级：

A. 增加 Discover 模式；  
B. 增加 Refine 模式；  
C. 引入 PostGIS；  
D. 引入 Supabase Queues；  
E. 增加音乐和时间切片；  
F. 增加照片、打卡和游后纪念；  
G. 准备国内部署与收费。

## 阶段成果

得到一版可以持续被真实使用的私人 Beta。

## 验收标准

- 至少 5 名真实用户完成路线生成；
- 至少 3 条路线被实际走完；
- 核心流程无阻塞性错误；
- 关键 API 成本可控；
- 已有明确的下一阶段数据依据；
- 不凭主观想象提前扩张功能。

# 总体里程碑

| 阶段 | 可交付价值 |
|---|---|
| 1 | 可访问、符合设计风格的网站骨架 |
| 2 | 可登录、保存和分享 |
| 3 | 可生成真实地图路线 |
| 4 | 可智能推荐沿途地点 |
| 5 | AI 可协作规划并增加历史文学内容 |
| 6 | 可供亲友真实使用的私人 Beta |

---

# 推荐优先级

必须优先：

1. 路线真实可走；
2. 地点真实存在；
3. 时间轴正确；
4. 沿途候选点有价值；
5. 手机端途中可用；
6. AI 失败时仍可使用。

可以稍后：

- 长篇故事；
- 音乐播放；
- 时间切片；
- 复杂成就；
- 公共社区；
- 收费系统；
- 国内部署；
- 高级空间数据库。

---

# 开发完成定义（Definition of Done）

一个功能只有在以下条件全部满足时才算完成：

- 桌面和手机端均可用；
- 正常、加载、空、失败状态齐全；
- 有输入校验；
- 有权限校验；
- 有错误日志；
- 外部 API 有超时；
- 不泄露密钥；
- 关键逻辑有测试；
- 通过至少一条真实城市路线验证；
- 文档和 migration 已更新。
